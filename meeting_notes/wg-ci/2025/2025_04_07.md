* Topic: Release process and Trilinos integration testing
  * Review and discuss revising of current processes

**Release sequence of events**
```
1. Cut release-candidate (rc) branch, announce for early testing
2. Repeat until rc complete (no more rc changes and testing passes):
   + Updates to rc
   + Changelog review
   + Ongoing nightly integration testing
   + Pre-release PR to Trilinos
3. Finalize "release" branches: merge-rc-to-master
4. Snapshot merge-rc-to-master release branches to Trilinos
   - Currently semi-broken: squash-merge of master removes the snapshotted SHA, spack version outdated after Trilinos update
5. Submit PRs
6. Following merge of release PRs (Kokkos, Kokkos Kernels, Trilinos):
   * Tag and release artifacts ("release!")
   * Update develop (changelog and readme)
   * Update spack packages
```

**Possible places for improvement**
```
* Drop master - eliminates "event 3" as blocker and following repeated tasks (i.e. rerun all testing)
* Reduce time of "event 2" (largest bottleneck)
* Snapshot to Trilinos after merge-rc-to-master complete
  * Would require changing requirement that Trilinos PR is in ready-to-merge state as part of release (event 5), in favor of passing pre-release PR testing (event 2)
  * Would help fix some current issues (snapshot SHA, spack version mismatch)
  * Potential for out-of-sync state of package vs release
```
